# Контекст проекта: Wordsling

**Суть:** Система Speech-to-Text на базе Gboard.

**Текущий фокус:** Rust-серверная часть проекта — реализация текстового процессора (FrankenLab).

## Архитектура
1. **Клиент (Android, Kotlin):** Открывает текстовое окно с клавиатурой Gboard. Вводимый/распознанный текст сериализуется и отправляется по TCP-сокету.
2. **Сервер (Rust) [ТЕКУЩИЙ ФОКУС]:** Читает данные из TCP-сокета, обрабатывает их и направляет в сфокусированное текстовое поле на хосте.

## Текущее состояние пайплайна Wordsling server

Пайплайн полностью собран и работает end-to-end:

```
TcpServer -> Lexer -> FrankenLab -> ScreenWriter
```

Плюс независимый `UserActivityTracker` (стаб).

### TcpServer (`src/tcp_server.rs`)
- Принимает TCP-подключения от телефона.
- Переживает переподключения.
- Корректно собирает UTF-8 при разрывах многобайтных символов между TCP-пакетами.
- Передает декодированные строковые чанки в Lexer через `mpsc::channel`.
- Предоставляет write-интерфейс для отправки команд обратно на телефон.

### Lexer (`src/core/lexer.rs`)
- Полноценная реализация, покрыта тестами.
- Конвертирует поток текстовых чанков в поток лексем (`LexemeTransfer`).
- Корректно обрабатывает фрагментацию TCP, erase-команды, стабилизацию.
- Выполняет декапитализацию всех слов.

### FrankenLab (`src/core/text_processor.rs` и `surgical_table.rs`)
- **Текущее состояние:** Реализован скелет реактивного автомата (`SurgeTable`).
- Хранит сырой текст (`_cutting_board`) и экранное отражение (`_franken_board`) в виде `Vec<char>`.
- Учитывает стабилизацию Gboard (`_stabilization_anchor`), запрещая стирать зафиксированный текст.
- Аккумулирует команды для ScreenWriter во внутренней очереди (`_pending_transfers`).
- Готов к внедрению подстановок через маппинг индексов (`comb`).

### ScreenWriter (`src/core/screen_writer.rs`)
- Получает команды `ScreenTransfer` от FrankenLab.
- Выводит текст в сфокусированное окно через clipboard paste (`Ctrl+V`).
- Дебаунсинг: накапливает фрагменты текста и вставляет их одной операцией
  после истечения cooldown-интервала.
- Backspace: сначала съедает локальный буфер, затем посылает реальные нажатия.
- **Clipboard preservation:** перед первой подменой буфера обмена сохраняет
  пользовательский текст и восстанавливает его после завершения серии вставок.

### UserActivityTracker (`src/core/user_activity_tracker.rs`)
- Стаб. Поток ничего не делает.

## Тестовый стенд

В `src/core/core_tests.rs` (и `lexer_plus_text_processor.rs`) находится отладочный мини-пайплайн:
```
chunks -> Lexer -> FrankenLab -> fake screen sink -> final visible text
```
- Подает текстовые чанки в формате протокола (включая сложные трейсы реальной диктовки).
- Прогоняет их через `Lexer` + `FrankenLab`.
- Собирает `ScreenTransfer` в памяти и применяет их к строковому буферу ("виртуальный экран").
- Проверяет финальный текст ассертами. Стенд успешно проходит проверку базовой стабилизации.
- Модуль временный, будет удалён после завершения отладки.

## Протокол обмена (Wordsling Protocol)

**Логика работы:**
Клиент (Android) непрерывно транслирует состояние ввода Gboard.
Передача идет "на лету", включая сырой текст, корректировки и фиксации.

**Специальные маркеры (inline):**
1. **`[N]` (Backspace):** Команда удаления `N` символов. Генерируется,
   когда Gboard меняет распознанное слово "задним числом".
2. **`*` (Commit/Stabilization):** Признак фиксации. Gboard определил,
   что распознавание фрагмента завершено. Этот кусок текста больше
   корректироваться не будет.

**Транспортный уровень (TCP):**
Данные передаются непрерывным потоком байт (raw stream) без разделителей кадров.

## Принятые технические решения
1. **Вывод текста:** вставка через clipboard (`Ctrl+V`) с дебаунсингом.
2. **Удаление символов:** эмуляция Backspace через `SendInput`.
3. **Clipboard preservation:** сохранение и восстановление пользовательского
   содержимого буфера обмена между сериями вставок.
4. **Кроссплатформенность:** сейчас Windows, долгосрочная цель — Linux.
   Windows-специфику изолировать через `#[cfg]`.

Дальнейшее развитие текстового процессора FrankenLab (на базе `SurgeTable`):

- Реализация скользящего окна (pruning), чтобы буферы `_cutting_board` и `_franken_board` не переполняли память.
- В перспективе: капитализация, команды действий (голосовые забои, выходящие за пределы Gboard-стабилизации).
