# Контекст проекта: Wordsling

**Суть:** Система Speech-to-Text на базе Gboard.

**Текущий фокус:** Rust-серверная часть проекта, принимающая поток ввода с телефона и перенаправляющая его в текстовое поле на хосте.

## Архитектура
1. **Клиент (Android, Kotlin):** Открывает текстовое окно с клавиатурой Gboard. Вводимый/распознанный текст сериализуется и отправляется по TCP-сокету.
2. **Сервер (Rust) [ТЕКУЩИЙ ФОКУС]:** Читает данные из TCP-сокета, обрабатывает их и направляет в сфокусированное текстовое поле на хосте.

## Протокол обмена (Wordsling Protocol)

**Логика работы:**
Клиент (Android) непрерывно транслирует состояние ввода Gboard. Передача идет "на лету", включая сырой текст, корректировки и фиксации.

**Специальные маркеры (inline):**
1. **`[N]` (Backspace):** Команда удаления `N` символов. Генерируется, когда Gboard меняет распознанное слово "задним числом".
   *Пример:* `ехал грек чер` + `[4]а через реку` = `ехал грека через реку`.
2. **`*` (Commit):** Признак фиксации. Gboard определил, что распознавание фрагмента завершено (например, после паузы). Этот кусок текста больше корректироваться не будет.

**Обычный текст:**
Всё, что не является маркерами `[N]` или `*`, передается "как есть" в кодировке UTF-8, включая пробелы и пунктуацию.

**Транспортный уровень (TCP):**
Данные передаются непрерывным потоком байт (raw stream) без разделителей кадров. Сервер читает поток по мере поступления и самостоятельно обрабатывает возможные разрывы многобайтных UTF-8 символов на границах TCP-пакетов.

## Принятые технические решения
1. **Основная стратегия вывода текста в поле ввода:** использовать **вставку через clipboard**.
   Причины:
   - это быстрее, чем поштучная эмуляция клавиатуры;
   - это надежнее на практике;
   - такой подход уже успешно использовался в Hobot.
2. **Стратегия удаления символов:** для обработки `[N]` использовать **эмуляцию Backspace**.
3. **Fallback-стратегия:** если clipboard-вставка окажется недостаточной для отдельных приложений/полей, можно вернуться к идее прямой клавиатурной эмуляции как запасному варианту.
4. **Кроссплатформенность:** сейчас работа идет под Windows, но долгосрочная цель — возможность возвращения на Linux. Поэтому Windows-специфику желательно изолировать, а где возможно — оставлять stubs/fallbacks для Linux.
