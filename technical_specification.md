# Wordsling Technical Specification

## 1. Purpose of the document

This document describes the intended end-to-end architecture of Wordsling.
Its main goal is to preserve the big picture of the system before all parts are fully implemented.

At the current stage, the most important objective is not to freeze every low-level detail,
but to clearly frame the processing pipeline, the roles of its stages, and the core invariants.

The document is expected to grow over time. In particular, it should later include:
- a more complete description of the Android side;
- text substitution mechanisms;
- action commands such as capitalization control or language switching;
- exact punctuation lists and other normalization rules.

---

## 2. System overview

Wordsling is a continuous speech-to-text pipeline built around Gboard on Android.
The Android device acts as a speech input source, while the host PC receives the recognized text
and injects it into the currently focused input field.

The processing chain is:

1. **Android client / Gboard capture**
2. **Serialization into a TCP byte stream**
3. **TCP server on the host** (`TcpServer`)
4. **UTF-8 text assembly from the raw byte stream** (part of `TcpServer`)
5. **Lexeme parser** (`Lexer`)
6. **Post-lexical text processing** (`FrankenLab`)
7. **Screen writer** (`ScreenWriter`)

An independent `UserActivityTracker` runs in a separate thread (currently a stub).

Each stage runs in its own thread and communicates via `mpsc` channels.
Shutdown is cascading: when the TCP server stops, it drops its sender,
which causes each downstream stage to exit its blocking `recv()` loop in sequence.

This separation is important because the incoming speech stream is dynamic:
Gboard may revise previously recognized text, emit stabilization markers, and request deletion
of part of the recently produced unstable text.

## 3. Input protocol generated from Gboard output

### 3.1. General idea

Gboard generates ordinary user-facing text:
- words;
- whitespace;
- punctuation.

To transmit that text in a revisable streaming form, the Android side serializes it with two service constructs:

- `*` — **stabilization marker**;
- `[n]` — **erase command**, where `n` is the number of characters to delete.

There are no other service constructs in the protocol.

### 3.2. Meaning of service constructs

#### Stabilization marker: `*`

`*` means that all text received before this marker is now considered stable.
That text will no longer be revised by future backspace commands generated by Gboard.

In later processing stages, this marker allows the system to:
- flush stable text to the target window;
- forget already committed text;
- keep only the unstable tail in the working buffer.

#### Erase command: `[n]`

`[n]` means that `n` characters must be deleted from the unstable part of the already processed stream.
This construct is used when Gboard revises its recent recognition result.

### 3.3. Important protocol assumption

The Android device is intended to behave as a microphone-only frontend.
If technically feasible, manual typing on the phone should be disabled.

This assumption simplifies the protocol significantly:
- `*`, `[` and `]` are treated as service syntax, not ordinary text;
- no escaping mechanism is required at this stage.

## 4. Transport and text assembly

The transport layer is a raw TCP byte stream without message framing.

The host-side TCP server is responsible for:
- accepting connections from the Android device;
- surviving reconnects;
- reading a continuous stream of bytes;
- correctly assembling UTF-8 text even if a multibyte character is split across TCP packet boundaries;
- forwarding decoded `String` chunks to the next stage.

A crucial consequence of TCP streaming is that **text may be split at any position**.
The system must not assume that a whole word arrives in one chunk.

This applies to all ordinary text.
For example, one word may arrive as:
- `hel`
- `lo`

and must still be treated as one logical word.

### Important note about newlines

Chunk boundaries are not represented by line breaks.
If `\n` appears in the stream, it is a real whitespace character generated by Gboard,
not an artifact of transport fragmentation.

### Implementation status

Fully implemented and tested.

## 5. Lexeme parser

### 5.1. Role of the lexer

The lexeme parser is the first semantic processing stage after UTF-8 text assembly.

It receives a stream of decoded text chunks and converts them into a stream of lexemes.
Its task is to separate:
- word content;
- word boundaries;
- whitespace;
- punctuation;
- service protocol events.

The lexer must not perform high-level text rewriting.
It only identifies and emits semantic elements of the input stream.

### 5.2. Execution model

The lexer runs in its own worker thread.
It:
- blocks on the receiver connected to the TCP server;
- processes incoming text immediately;
- emits lexemes into its own output channel;
- avoids any artificial delays.

The lexer is a state machine.
It preserves state across chunk boundaries, especially when:
- a word is split across chunks;
- a service construct like `[n]` is received only partially.

### 5.3. Core lexer invariants

The lexer satisfies the following invariants:

1. **No artificial delays**
   - The lexer does not slow down the stream with `sleep()` or similar timing logic.

2. **Preservation of significant text content**
   - Significant text content means the ordinary text that Gboard produces for the user:
     words, punctuation, and whitespace.
   - Service constructs such as `*` and `[n]` are not part of that visible text.
   - The lexer does not drop, invent, reorder, or duplicate significant text characters.

3. **Chunk-boundary transparency**
   - TCP chunk boundaries do not affect the semantic result.
   - A word split between chunks is still recognized as one logical word.

4. **Protocol extraction**
   - Service constructs are extracted from the text stream and emitted as separate lexemes.

### 5.4. Word model

A word is defined negatively:
- anything that is **not** whitespace;
- **not** punctuation;
- **not** a service construct.

Additional rules:
- words may contain digits;
- apostrophe (`'`) is not treated as punctuation and therefore doesn't split words.
  For example "isn't" (Gboard sometimes generates it without spaces);
- hyphen (`-`) is treated as punctuation and splits words
  (e.g., compound words like `word-word` become two words separated by a hyphen).

### 5.5. Punctuation list

The lexer uses a strictly predefined set of characters as punctuation.
Based on Gboard's output behavior, the following characters form the exact punctuation list:
- `.` (dot)
- `,` (comma)
- `;` (semicolon)
- `:` (colon)
- `?` (question mark)
- `!` (exclamation mark)
- `#` (hash / number sign)
- `/` (slash)
- `'` (apostrophe)
- `@` (at sign)
- `-` (hyphen)

Any character that is not in this list, is not a whitespace, and is not a service construct
(`*`, `[`, `]`) is automatically treated as part of a word.

### 5.6. Decapitalization

Gboard is not reliable enough in capitalization.
Therefore, the lexer normalizes word content to lowercase.

Capitalization will later be handled by the post-lexical processing stage (FrankenLab).
This gives the system a single and predictable place for capitalization rules,
instead of relying on Gboard's inconsistent behavior.

### 5.7. Lexeme model

The lexeme model is event-like.
Word boundaries are represented explicitly, and word content is emitted separately.

The lexer emits the following kinds of lexemes:

```rust
pub enum LexemeTransfer {
    // Phone events
    WordStart,
    WordPart(String),
    WordEnd,
    Whitespace(char),
    Punctuation(char),
    EraseStart,
    BackspaceCount(u32),
    EraseEnd,
    Stabilization,

    // User events
    UserActivityDetected,
}
```

`UserActivityDetected` is emitted by the `UserActivityTracker` (not by the lexer)
and injected into the same channel to notify FrankenLab about user input activity on the host.

### 5.8. Meaning of word lexemes

The word-related lexemes have the following meaning:

- `WordStart` — a word begins here;
- `WordPart(String)` — a fragment of a word;
- `WordEnd` — the word ends here.

This representation is deliberately stream-oriented.
It allows the parser to emit meaningful events even if the transport cuts the word into fragments.

Example:
- incoming chunks: `hel` and `lo `
- lexer output:
  - `WordStart`
  - `WordPart("hel")`
  - `WordPart("lo")`
  - `WordEnd`
  - `Whitespace(' ')`

### 5.9. Meaning of erase lexemes

The erase command is intentionally decomposed into a small sequence of lexemes:
- `EraseStart`
- `BackspaceCount(n)`
- `EraseEnd`

This preserves the event nature of the stream and keeps the protocol visible to the next stage.
It also matches the state-machine nature of the lexer.

### 5.10. Example of lexeme extraction

Example dictated phrase:

`Имею я златые горы.`

Possible serialized protocol stream:

`Имею ли я [5]я златые горы.*`

One possible arrival pattern:
- `Имею ли я [5]`
- `я зла`
- `тые горы.*`

A corresponding lexeme stream may look like:
- `WordStart`
- `WordPart("имею")`
- `WordEnd`
- `Whitespace(' ')`
- `WordStart`
- `WordPart("ли")`
- `WordEnd`
- `Whitespace(' ')`
- `WordStart`
- `WordPart("я")`
- `WordEnd`
- `Whitespace(' ')`
- `EraseStart`
- `BackspaceCount(5)`
- `EraseEnd`
- `WordStart`
- `WordPart("я")`
- `WordEnd`
- `Whitespace(' ')`
- `WordStart`
- `WordPart("зла")`
- `WordPart("тые")`
- `WordEnd`
- `Whitespace(' ')`
- `WordStart`
- `WordPart("горы")`
- `WordEnd`
- `Punctuation('.')`
- `Stabilization`

The example illustrates two important facts:
- transport fragmentation may split a word into multiple `WordPart(...)` lexemes;
- stabilization and erase commands are emitted as separate semantic events.

Fully implemented and tested.

## 6. Post-lexical text processing (FrankenLab)

The post-lexical text processor (internally named `FrankenLab`) consumes lexemes
and produces screen transfer commands for the screen writer.

### 6.1. Responsibilities

FrankenLab is expected to:
- maintain the distinction between stable and unstable text;
- apply backspace commands to the unstable tail;
- decide when text is ready to be flushed to the output layer;
- restore proper capitalization;
- fix spacing and punctuation details when necessary;
- perform future text substitutions;
- interpret future action commands such as "capitalize", "switch language",
  and similar directives.

This stage is where the system becomes more than a transport bridge.
It will gradually evolve into the main text-intelligence layer of Wordsling.

### 6.2. Output contract

FrankenLab sends `ScreenTransfer` commands to the screen writer:

```rust
pub enum ScreenTransfer {
    Text(String),
    Backspace(usize),
}
```

### 6.3. Implementation status

FrankenLab has been refactored into a reactive state machine built around an internal
`SurgeTable`. The `SurgeTable` maintains two separate character buffers (`cutting_board` for
raw Gboard input and `franken_board` mirroring the host's screen).

It currently enforces Gboard stabilization anchors (`*`) and processes backspace commands safely
using character-level vectors (`Vec<char>`). While it successfully manages the text stream and
erasure limits, text substitution logic is still pending.

The next development focus is implementing text substitutions and user-driven
history manipulation (e.g., custom voice erasures).

## 7. Screen writer

The screen writer is the final execution layer.
Its job is not to parse or interpret the stream, but to apply already prepared output operations
to the currently focused input field on the host.

### 7.1. Text output mechanism

Text is output via clipboard paste (`Ctrl+V`). A debouncing mechanism is applied:
incoming text fragments are accumulated in a local buffer and pasted as a single
operation after a cooldown interval elapses. This prevents clipboard race conditions
where the target application reads stale clipboard content.

### 7.2. Backspace mechanism

Backspace commands are first applied to the local buffer (free, no OS delay).
Only if the buffer is exhausted, real `send_backspace()` keystrokes are emitted to the OS.

### 7.3. Clipboard preservation

The user's original clipboard content is saved before the first paste in a session
and restored after the last paste completes. This ensures that dictation does not
destroy clipboard data the user copied earlier.

Limitation: only plain text clipboard content is preserved.

### 7.4. Timing constants

- `_COOLDOWN_MS` (100 ms) — minimum interval between consecutive `Ctrl+V` operations
  and the delay before restoring the clipboard after the last paste.
- `_CLIPBOARD_ASSIMILATION_MS` (20 ms) — minimum delay between updating the clipboard
  and pressing `Ctrl+V`, to let the OS process the clipboard update.
- `_BACKSPACE_DELAY_MS` (0 ms) — delay between consecutive backspace keystrokes.
  Currently zero because the OS input queue serializes keystrokes safely.

### 7.5. Implementation status

Fully implemented.

## 8. Main design direction

The main architectural idea of Wordsling is the separation of concerns:
- transport deals with bytes and UTF-8;
- the lexer deals with semantic segmentation of the stream;
- FrankenLab deals with meaning, normalization, corrections, substitutions, and actions;
- the screen writer deals with final host-side output.

This structure should make the system easier to reason about, extend, and debug.
It also gives a solid foundation for future features without requiring redesign
of already defined stages.

---

## 9. Open points to refine later

The following details are intentionally left flexible for now:
- exact normalization rules for special characters;
- the detailed structure of post-lexical processors inside FrankenLab;
- Android-side implementation details;
- the exact representation and routing of future action commands.

These are important, but they should not block the definition of the main architecture.
